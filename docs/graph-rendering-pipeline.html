Pipeline Overview

The pipeline has 8 distinct phases. Data enters as a raw JSON API payload and exits as positioned, styled Vue Flow nodes and edges rendered in an SVG canvas. Hover any stage in the diagram below for a description.

01

Server API & Data Fetching

The pipeline begins when GraphHydrator.assembleGraphData() fires. It tries /graph/summary first; if that returns non-200 it falls back to /graph. Both endpoints return the same shape — a { packages: GraphApiPackagePayload[] } JSON envelope.

Request Flow

// GraphHydrator.ts — simplified
const cacheKey = `v5-external-deps-by-name:${this.baseUrl}`;
if (this.cache.get(cacheKey)) return cachedData; // in-memory GraphDataCache

const res = await fetch('/graph/summary', { signal: AbortSignal.timeout(30_000) });
const graph = this.hydratePackages(payload.packages, new EntityRegistry());
this.cache.set(cacheKey, graph); // prevents re-fetch on hot reload

Cache Version Coupling

The cache key is v5-external-deps-by-name. If you add a new field to the payload, bump this version string in GraphHydrator.ts:82 or all existing browser caches will return stale data silently. There's no TTL — the cache lives for the duration of the page session (in-memory) or until manually cleared.

Module Details on Demand

When a user drills down into a specific module, fetchModuleDetails(moduleId) fires a separate /module-details?moduleId=… request. This response is also cached in moduleDetailsCache — a plain Map<string, Module | null>. The null sentinel prevents refetching 404'd modules.

02

Entity Hydration

Raw JSON is deserialized into a rich class hierarchy by GraphHydrator's private hydrateX() methods. This is not simple JSON.parse — it constructs typed instances with methods, enforces invariants via EntityRegistry, and handles multiple field name variants for resilience against API drift.

  • Package Pass 1 — entity construction
    For each raw package: hydrateModules → hydrateClasses/Interfaces/Imports → construct Package instances. EntityRegistry.register(id, entity) deduplicates: if the same UUID appears twice (e.g., an interface referenced by two classes), the registry returns the same object.
  • Package Pass 2 — dependency cross-linking
    After all packages exist in the byId map, iterate again and populate pkg.dependencies / devDependencies / peerDependencies Maps. This two-pass approach avoids forward-reference errors.
  • Field name normalization
    The hydrator checks multiple field names: typeAliases ?? type_aliases, import path ?? relativePath ?? fullPath ?? name. This handles both camelCase API responses and snake_case DB column names without a transformation layer.
  • External deps scope hydration
    hydrateExternalDepsByName() copies only the three valid scope values into a plain record. Any unknown scope string from the DB is silently dropped. This record is stored on Package.externalDepsByName and later used by buildExternalPackageKindLookup() in the edge builder.
TypeCollection<T> Pattern

Throughout the pipeline, collections can be Array<T>, Map<string, T>, or Record<string, T>. The typeCollectionToArray() and mapTypeCollection() utilities normalize all three forms. Always use these — never assume a collection is an array.

04

Sugiyama Layout Algorithms

buildOverviewGraph() in buildOverviewGraph.ts implements a six-pass Sugiyama framework. This is where the topological structure of the codebase is encoded as graph layout data. No pixel positions yet — just rank weights, integer layer indices, and sort orders.

Sugiyama Framework

The Sugiyama framework for DAG drawing works in three conceptual steps: (1) cycle breaking — mark back edges so the graph is temporarily a DAG; (2) layering — assign each node an integer "rank" representing its column; (3) crossing minimization — reorder nodes within each layer to reduce edge crossings. This codebase implements all three plus a centrality refinement and a recursive folder-level re-run.

  • DFS Back-Edge Detection
    detectBackEdges() performs iterative DFS tracking a "on-stack" set. Any edge pointing to an ancestor on the current DFS path is a back-edge — the closing edge of a cycle. Back-edge IDs are collected into a Set<string> and excluded from layering and barycenter passes. Back edges are marked with data.isBackEdge = true for dashed styling.
  • Fan-In Weighted Longest-Path Layering
    computeModuleLayoutWeights() assigns a weighted depth to each node using memoized DFS. The step cost to traverse an edge is 1 + log₂(max(1, fanIn(target))). A module imported by 8 others has step cost 4 instead of 1 — it creates larger horizontal gaps in the layout, visually emphasizing foundational modules. layoutWeight = -weightedDepth so foundations (depth 0) are leftmost.
  • Integer Layer Assignment
    computeLayerAssignment() is a separate simpler pass that produces integer layer indices (0, 1, 2, …) using longest-path with uniform step cost 1. This drives column assignment in the position stage. Conceptually separate from the weighted depth which controls spacing within columns.
  • Barycenter Crossing Minimization
    computeBarycenterSortOrder() runs 4 alternating sweeps (2 L→R, 2 R→L). In each sweep, a node's position within its layer is set to the weighted average position of its neighbors in adjacent layers. The weight is 1/distance — neighbors 1 layer away have full weight, those 2 away have half, etc. This distance-weighted variant handles "long edges" (43% of edges span >1 layer) that the standard adjacent-only approach misses.
  • Centrality Pyramid Re-order
    After barycenter, within each layer where max-degree > 2× mean-degree (CENTRALITY_GATE = 2), high-degree nodes are moved toward the vertical center using a spiral outward pattern: position index 0→mid, 1→mid+1, 2→mid-1, 3→mid+2, … This places widely-used hubs (e.g., a vue or consola external package) at the visual trunk, shortening average edge length.
  • Folder Clustering + Cross-Folder Edge Lifting
    clusterByFolder() groups modules by directory path and creates group type parent nodes. liftCrossfolderEdgesToFolderLevel() then replaces module→module edges that cross folder boundaries with three-segment representations: a source stub (module → source folder boundary), a trunk (folder → folder, type crossFolder), and a target stub (target folder boundary → module). This preserves routing information while aggregating count.
  • Folder-Level Sugiyama (Second Pass)
    applyFolderLevelSugiyama() re-runs the entire module-level pipeline (back-edge detection → weighted depth → layer assignment → barycenter) but this time on just the folder nodes and their cross-folder trunk edges. This corrects folders whose aggregated rank would be misleading (e.g., src/ contains both a shallow server.ts and deep main.ts — the aggregation would pick the wrong position). Only folders that participate in a cross-folder edge are overridden.

Parallel Edge Bundling

bundleParallelEdges() is skipped for graphs under 50 edges (overhead not worth it). For larger graphs it groups by source\0target key, picks the highest-priority edge kind as the representative, and records all bundled types in data.bundledTypes. This reduces SVG DOM element count by 20–40% in real codebases. Priority order is defined in edgePriority.ts.

05

Position Assignment

computeSimpleHierarchicalLayout() in simpleHierarchicalLayout.ts is a pure function that converts the Sugiyama layout metadata (layerIndex, sortOrder) into pixel coordinates. It also sizes parent containers.

Two-Phase Layout

  • Child Layout (within folders)
    Single-column mode: when a folder has no intra-folder edges, all children share the same X and stack vertically by sortOrder. Gap = 200px. This is the common case — siblings in a directory are peers, not a dependency chain.

    Layer-column mode: when intra-folder edges exist (modules in the same directory import each other), children are grouped by layerIndex into sub-columns exactly like root-level layout. Foundations-left, consumers-right.
  • Parent Sizing
    Parent width = max(220, childWidth + 200px padding). Parent height = max(80, 100px header + contentHeight + 100px bottom). These explicit sizes are written as CSS pixel strings to node.style.width and node.style.height — the same format Vue Flow's internal handleParentExpand uses, ensuring consistent dimension tracking via measured field sync.
  • Root Node Column Layout
    Root nodes (no parentNode) are grouped by layerIndex, sorted within each layer by sortOrder descending (higher = top), then placed in columns. Column X advances by max(nodeWidths in column) + 120px gap. Column Y advances by nodeHeight + 120px gap.
Layout Constants Are Hardcoded

ROOT_H_GAP = 120, ROOT_V_GAP = 120, CHILD_GAP = 200 are hardcoded literals in simpleHierarchicalLayout.ts. The theme object at graphTheme.layout.spacing has { horizontal: 40, vertical: 60 } — these values are not wired to the layout engine. The comment in graphTheme.ts:218 literally says "The layout engine is not wired to this object yet." This means changing theme spacing has zero effect on layout.

Pre-measurement & Measurement-Driven Relayout

Before layout runs, useNodePremeasure renders each node into a hidden off-screen container (display:contents) to get real pixel dimensions. These are applied before computeSimpleHierarchicalLayout runs, so the layout has accurate sizes rather than the estimated fallbacks (CHILD_ESTIMATED_WIDTH = 320, CHILD_ESTIMATED_HEIGHT = 260).

After Vue Flow renders nodes into the DOM, ResizeObserver events trigger scheduleMeasurementDrivenRelayout() which runs inside a requestAnimationFrame callback. This handles nodes whose real size differed significantly from the pre-measured estimate, causing a second layout pass with correct dimensions. The loop is guarded by change detection (Math.abs(current - measured) > 1px) to prevent infinite relayout cycles.

Manual Offset Overlay

After layout, graphStore.applyManualOffsets() adds accumulated { dx, dy } offsets to node positions. These come from collision-resolution pushes. Offsets are stored in manualOffsets: Map<string, ManualOffset> and survive re-layouts via mergeManualOffsets() which accumulates (not replaces) deltas. This means user-adjusted positions are sticky across layout refreshes triggered by settings changes.

06

Node Rendering

Vue Flow dispatches to custom node components based on the node.type string. All nodes ultimately compose BaseNode.vue which provides the shared chrome (header, badge, handles, diagnostics strip).

ModuleNode.vue

The most complex node. Renders: metadata properties section, external dependencies (package → symbol list), embedded class cards (compact mode), entity list (functions, types, enums, const/vars). All sections are in collapsible CollapsibleSection wrappers. Dependency level drives module-node--high-deps / module-node--critical-deps CSS classes.

GroupNode.vue

Folder container. Handles collapsed/expanded state. When data.isCollapsed is true, renders a compact summary showing child count. The group node positions its border with overflow: visible and padding: 0 — child positions are relative to the wrapper, not the visible border.

SymbolNode.vue

Used for standalone class/interface nodes in graph mode. Renders member lists (properties and methods) and connection handles for extends/implements edges.

ExternalPackageNode.vue

Represents an npm package imported by the workspace. Shows total symbol count across all modules that import it. Always a leaf in the dependency direction.

Handle Routing

Every node has Vue Flow Handle components for edge connection points. Regular module/symbol nodes use sourcePosition / targetPosition from the layout direction (always LR → right/left). Folder group nodes additionally expose named handles: folder-right-out, folder-left-in, folder-right-stub, folder-left-stub — used by the cross-folder edge split system.

Content-Visibility Culling

After layout completes, graphRootRef.value?.classList.add('cv-ready') is called. This triggers a CSS rule that enables content-visibility: auto on nodes — the browser skips rendering nodes outside the viewport entirely. This is skipped on Firefox due to "severe toggle-thrash regressions" noted in the code comment at useGraphLayout.ts:427.

Node Data Flow: Sorting

ModuleNode uses five sorting functions from moduleNodeSorting.ts: sortNodeProperties, sortExternalDependencies, sortEmbeddedSymbols, sortModuleEntities, sortSectionsByTitle. These run inside computed() so they only re-execute when reactive data changes.

Custom Edge Geometry Engine

Rather than relying solely on Vue Flow's built-in path helpers, the codebase ships a complete custom geometry engine in edgeGeometryPolicy.ts. This engine is responsible for producing orthogonal polylines with rounded corners for folder-system edges.

  • Handle Side Inference
    getHandleSide(handleId) parses the handle ID string (e.g., folder-right-out → 'right', relational-in → 'left'). When the handle ID doesn't encode a direction, inferHandleSide(from, to) uses the dominant axis of the source→target vector, with ties going horizontal.
  • Stub Extrusion
    buildEdgePolyline() extrudes short stubs from source and target points along their side normals. Group nodes (folders) get a short GROUP_ENTRY_STUB_PX = 12 stub; regular module nodes get NODE_PRE_APPROACH_STUB_PX + NODE_FINAL_APPROACH_PX = 24px with a two-segment approach (a longer pre-approach then a final close approach) that prevents the arrowhead from touching the node border.
  • Orthogonal Midpoint Insertion
    insertOrthogonalMidpoints() detects whether the free segment between source and target stubs is diagonal. If so, it inserts: an L-shape midpoint (single corner) when source exits horizontally and target enters vertically (or vice versa), or an S-shape dogleg (two midpoints through the horizontal/vertical midline) when source and target share the same exit axis. This ensures all intermediate segments are strictly axis-aligned.
  • Rounded Corner Path Generation
    buildRoundedPolylinePath() iterates the polyline and replaces each interior corner with a quadratic Bézier arc. The effective radius at each corner is clamped to min(cornerRadius, incomingLength/2, outgoingLength/2) — preventing arcs from overshooting into adjacent segments. The output is a { start, segments: (line | quadratic)[] } structure that callers convert to SVG path data.

Edge Hover System

Hovering a node calls applyHoverEdgeHighlight() which iterates all edges connected to that node. Connected edges get the EDGE_HOVER_CLASS = 'edge-hover-highlighted' CSS class and a --edge-hover-base-stroke CSS custom property injected into their inline style. CSS rules in the graph stylesheet use this variable to animate the stroke color. Disconnected edges get edge-selection-dimmed. The entire operation is batched into updateEdgesById(Map) for a single Pinia state update.

Selection Highlighting

useSelectionHighlighting applies three CSS class tokens to nodes: selection-target (the selected node), selection-connected (direct neighbors), selection-dimmed (everything else). Edges get edge-selection-highlighted or edge-selection-dimmed. Classes are applied/removed via getClassTokens() → Set<string> operations rather than string replacement, which handles class merging correctly.

08

Interaction & Runtime

Viewport Management

useGraphViewport wraps Vue Flow's panBy / zoomTo with custom wheel event handling. Mac trackpad vs. mouse wheel discrimination is done via wheelIntent.ts — it examines deltaMode, ctrlKey, and delta magnitudes to determine whether a wheel event is a pinch-zoom or a two-finger pan. The result drives either panBy() (trackpad) or zoomTo() (mouse).

Search

useGraphSearch produces SearchResult[]. useSearchHighlighting consumes them: matching nodes get full opacity, non-matching nodes get selection-dimmed. Edges connecting two non-matching nodes are hidden. The highlight state (searchHighlightState) is passed through to useSelectionHighlighting so that node selection during a search doesn't clear the search results.

Isolation Mode

useIsolationMode.isolateNeighborhood(nodeId) rewrites the graph completely — it builds a new node/edge set containing only the selected node and its neighbors, then calls processGraphLayout() on that subgraph. The original overview graph is saved to graphStore.overviewSnapshot. Returning calls restoreOverviewSnapshot(). The semanticSnapshot (pre-folder-transform nodes/edges) is used for neighborhood computation so filtering includes edges that may be hidden in the overview.

09

Drilldown Views

Symbol Drilldown

buildSymbolDrilldownGraph() focuses on a single class, interface, or module. It builds a star-shaped graph: the containing module at center, selected symbol(s) as children, and each symbol's properties/methods as leaf nodes. Symbol references from the module are wired as uses edges with dashed styling (strokeDasharray: '4 2'). Only references where both the source and target node exist in the graph are included.

Symbol Context Resolution

findSymbolContext() must scan the entire PackageGraph to find which module contains the selected class or interface. This is an O(packages × modules × symbols) scan that runs synchronously. For very large codebases this could be slow, but since it only runs once per drilldown interaction, it's acceptable in practice.

Module Drilldown

Focuses on a single module and its immediate import neighbors. Builds edges for direct imports, exports, and any symbol-level extends/implements relationships. Does not recurse — only depth-1 neighbors.

Expand-Collapse State Persistence

When isolation mode is entered, useExpandCollapseState saves the current expanded/collapsed state of each node's sections. When returning to overview, the state is restored. This prevents the UX jarring of sections collapsing when you return from a drilldown.

Identified Flaws

Issues found via code review. Severities: CRITICAL = data corruption or silent breakage; HIGH = observable wrong behavior; MEDIUM = technical debt with real consequences; LOW = minor inconsistency.

Critical
Non-unique edge IDs cause silent last-write-wins collision
src/client/utils/createGraphEdges.ts:201

createEdge() computes id: `${source}-${target}-${type}`. But the dedup map key is `${source}|${target}|${type}|${importName}`. When module A imports module B via two named imports (import { X } from … and import { Y } from …), two separate entries land in edgeMap with distinct keys but the same edge.id. Vue Flow uses edge.id as the React key — the second edge silently replaces the first in the rendered graph. Import names after the first are lost.

Make the edge ID unique: id: `${source}|${target}|${type}|${importName ?? ''}`. Alternatively, add a counter suffix for multi-import pairs. The deduplication key and the edge ID should be derived from the same formula.

Critical
DFS back-edge detection uses recursion — stack overflow on deep graphs
src/client/graph/buildOverviewGraph.ts:195

detectBackEdges() uses a recursive dfs(nodeId) inner function. JavaScript engines typically have a call stack limit of ~10,000 frames. A monorepo with deeply nested import chains (e.g., utility → helpers → primitives → …) can hit 50–100 levels deep easily in large codebases. At sufficient scale this throws a RangeError: Maximum call stack size exceeded and the entire graph initialization fails silently (caught by the outer try/catch in initializeGraph).

Replace the recursive DFS with an explicit stack: const stack = [startId]; while (stack.length) { const id = stack.pop(); … }. Track "on-stack" state using a Set that nodes are added to when pushed and removed when fully processed.

High
collapsedMeta.liftedEdgeCount is wrong for every folder
src/client/graph/cluster/collapseFolders.ts:127

After edge remapping, the loop stamps meta.liftedEdgeCount = totalLifted on every folder. totalLifted is the count of all edges remapped across all collapsed folders — not the count per-folder. A workspace with two collapsed folders (one with 3 lifted edges, one with 7) will show both folders as having 10 lifted edges instead of 3 and 7 respectively.

Track a Map<string, number> of per-folder lifted counts during the edge remapping loop, keyed by mappedSource and mappedTarget. Then stamp each folder's own count in the final loop.

High
graphTheme.layout.spacing is completely disconnected from layout
src/client/theme/graphTheme.ts:218

The theme object exposes layout.spacing = { horizontal: 40, vertical: 60, edge: 20, margin: 30 } but simpleHierarchicalLayout.ts uses hardcoded ROOT_H_GAP = 120, ROOT_V_GAP = 120, CHILD_GAP = 200. The theme values and layout constants are not only disconnected — they're also factually mismatched (120 ≠ 40). Any code that reads graphTheme.layout.spacing expecting it to reflect real layout behavior is silently wrong.

Either wire the layout engine to consume from graphTheme (import the theme and use its spacing values as constants), or remove graphTheme.layout entirely. Don't document future intent in source code that misleads consumers.

High
filterNodeChangesForFolderMode is a no-op stub
src/client/graph/buildGraphView.ts:15

The function signature implies different behavior based on folderModeEnabled, but both branches return changes unchanged. The _nodes parameter is also prefixed with underscore (intentionally unused). This function is exported and called from the interaction layer — any caller passing folderModeEnabled: true gets identical behavior to false. If folder-mode filtering was ever intended, it was never implemented.

Either implement the intended filtering logic or delete this function and remove all call sites. If the feature is deferred, add a TODO comment with a ticket reference — but don't ship a public function that lies about its behavior.

Medium
getModuleExports result is computed but never stored
src/client/utils/createGraphNodes.ts:267

getModuleExports(module) builds an array of export name strings and handles three different collection shapes (Map, Array, plain object). However, the returned value is never assigned to the module node's data. The moduleNode.data object has imports: getModuleImports(module) but no exports field. The computed export list is silently discarded every time a node is created.

Add exports: getModuleExports(module) to the module node data object, or delete getModuleExports entirely. The ModuleNode.vue template would need a new section to render it if added.

Medium
Sugiyama runs on folders with no cross-folder edges (wasted work)
src/client/graph/buildOverviewGraph.ts:707

applyFolderLevelSugiyama() correctly early-exits when folderEdges.length === 0, but aggregateFolderWeights() and liftCrossfolderEdgesToFolderLevel() still run their full iterations over all nodes and edges in graphs with no folder hierarchy. For small codebases this is negligible, but for large monorepos with thousands of modules it adds unnecessary O(n) passes.

Add a guard: if no modules have a parentNode (i.e., clustering was a no-op), skip all folder-level passes entirely.

Medium
getImportPath ignores all fallback fields
src/client/utils/createGraphNodes.ts:94

getImportPath() returns importValue.relativePath only — ignoring fullPath, path, and name. Every other path resolution in the codebase (including createGraphEdges.ts:346) uses the pattern imp.relativePath || imp.fullPath || imp.path || imp.name. If an import was stored with only a fullPath (which the DB schema allows), getModuleImports() returns empty strings for those entries, causing their edges to resolve against an empty path.

Align with the fallback chain: return importValue.relativePath || importValue.fullPath || importValue.path || importValue.name || undefined.

Low
Cache write guard doesn't distinguish graph identity
src/client/stores/graphStore.ts:98

The localStorage cache only skips writing when nodes exceed 1200 or edges exceed 5000. But there's no mechanism to invalidate the cache when the codebase changes (e.g., a new file is added). The cache key is v1:typescript-viewer-nodes — static. When the user restarts the server after adding files, the browser may render the old stale graph until the user manually clicks "clear cache".

Include a content hash (e.g., total module count + max created_at timestamp) in the cache key. The setCacheKey() action already exists for this purpose but receives null — wire it to a hash derived from the assembled graph.
